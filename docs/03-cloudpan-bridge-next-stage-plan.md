# CloudPan Bridge 下一阶段实施计划（修订版）

## 摘要

把当前控制台重构成“统一源端 -> 多目标端 -> 可选秒传/快传 -> 诚实降级补传”的主流程，并先解决两个高优先级问题：

- 删除 GitHub Actions 中会在 `push` 时自动构建/推送 Docker 镜像的链路
- 修正托管 OpenList 的体验闭环：当本机没有 `openlist/alist` 二进制时，不能再“点了像没反应”，而是必须先明确提示“是否拉取本机运行时”；如果用户不拉取，则明确引导只能改用远程/外部模式

## 关键改动

### 1. CI 风险项先收掉

- 修改 `.github/workflows/build.yml`，移除 Docker 镜像构建、登录、推送、导出 artifact 链路
- 保留 Python 基础校验、编译、测试和打包
- README 同步说明：Docker 仍支持本地手动构建/运行，但默认不再自动发布镜像

### 2. OpenList 模式重新定义

- 把当前混在一起的 `external/managed` 收成 4 个明确模式：
  - `external_local`：连接本机已有 OpenList
  - `external_remote`：连接远程已有 OpenList
  - `managed_binary`：本机二进制托管
  - `managed_docker_placeholder`：仅预留文案与状态位，本轮不真正实现自动起容器
- 页面和配置里明确拆开 4 套语义：
  - 当前连接实例的 URL / token / 用户名 / 密码
  - 托管运行参数
  - 托管初始化管理员
  - CloudPan Bridge 控制台管理员
- 保留旧配置兼容读取，但保存时写入新的分组结构

### 3. 托管 OpenList 缺失二进制时的强制引导

- 托管模式的启动流程改成固定三步：
  - 预检本机是否已有可用 `openlist.exe/alist.exe`
  - 没有则弹出明确选择
  - 只有满足前置条件后才允许真正启动
- 当本机没有二进制时，界面必须明确提示：
  - 当前未检测到本机 OpenList 运行时
  - 可选择“拉取本机二进制运行时”
  - 如果不拉取，则当前只能改用 `external_local` 或 `external_remote`
- 这条提示必须同时写到：
  - 页面状态区
  - 托管运行时详情
  - 日志
- 不允许再出现“点按钮但无日志、无错误、无进程”的静默失败体验

### 4. 本机托管运行时拉取策略

- 本轮只对 `managed_binary` 实现“拉取本机运行时”流程，不实现 Docker 托管
- 预期行为：
  - 页面先点“检测/启动托管”
  - 若未发现二进制，弹出“是否拉取本机 OpenList 二进制”
  - 用户确认后，后端负责下载到项目运行目录，例如 `.runtime/openlist`
  - 下载完成后再次预检并启动
- 如果用户拒绝拉取：
  - 当前模式不继续启动
  - 页面明确提示切到远程/外部模式
- 托管模式里“拉取运行时”和“连接已有实例”是两条不同链路，不能混淆

### 4.1 本机 Docker 环境安装与预检

- 把“本机 Docker 环境准备”加入正式实施项，而不是仅靠用户自行安装
- 本轮范围包括：
  - 检测本机是否已安装 Docker Desktop / Docker Engine 可用环境
  - 若未安装，在文档和控制台里明确给出安装引导与必需前置条件
  - 为 Windows 场景补充最小可执行安装建议、启动要求和验证步骤
- 本轮不要求项目内自动静默安装 Docker，但要求：
  - 页面和 README 能明确识别“本机无 Docker”
  - 能给出清晰的安装/启用提示
  - 安装完成后能重新检测并进入 Docker 相关预检链路
- 后续若真正落地 `managed_docker`，就基于这一步的环境预检结果继续推进

### 5. 托管初始化管理员策略

- 当前版本不再用 `admin + 空密码` 冒充初始化结果
- 本轮定成以下规则：
  - 托管初始化管理员属于单独配置项
  - 若用户未填写，系统可自动生成随机密码并提示保存
  - 若底层 OpenList/Alist 二进制不支持通过启动参数完成初始化，则必须明确提示：
    - 当前仅负责下载和启动进程
    - 不保证自动写入管理员账号密码
- 页面帮助提示必须明确区分：
  - OpenList 连接账号
  - 托管初始化管理员
  - 托管运行参数
  - 控制台管理员账号

### 6. 页面主流程重构

- Web 控制台主线收成 4 段：
  - 连接 OpenList
  - 接入源端
  - 配置任务
  - 执行与结果
- 保留 Tab 外壳，但每个 Tab 只承担单一职责，不再把关键配置散落在来回跳转的区域
- `overview` 变成真正的总任务面板，集中显示：
  - `source_path`
  - `target_key`
  - `target_path`
  - 当前 OpenList 模式与连接状态
  - 当前目标端能力提示
- 未完成控制台登录时，只显示登录区与必要说明，其余主操作区不应可交互

### 7. 源端接入流程收口

- 已挂载源：
  - 直接选挂载
  - 浏览目录
  - 写回 `source_path`
- 未挂载源：
  - 继续走 OpenList 挂载流
  - 但整合进主流程里，不再要求用户先跳到别处再回总览
- 保留现有这些能力：
  - 驱动列表
  - 驱动字段
  - 官方文档
  - 推荐默认值
  - 浏览器抓取登录态
- 复杂驱动继续保留接入说明与文档入口，但布局改成更清晰的辅助面板

### 8. 多目标端主线收正

- 目标端改成统一配置区，不再 Guangya 中心化
- 继续支持当前已实现目标端：
  - `guangya`
  - `openlist`
  - `localfs`
  - `webdav`
  - `s3`
  - `seafile`
  - `smb`
  - `ftp`
  - `sftp`
  - `azureblob`
- 目标端切换后动态显示：
  - 对应凭证字段
  - 目标端说明
  - 快传/秒传支持级别
- Guangya 相关专属文案和入口仅在 Guangya 目标端显示
- 继续诚实维持当前边界：
  - Guangya 是当前正式元数据秒传主实现
  - 其它目标端主要走普通上传/覆盖

### 9. 秒传/快传区改成目标端能力区

- 把“秒传区”改成“当前目标端快传/秒传能力区”
- 对不同目标端明确展示能力等级：
  - 支持元数据秒传
  - 仅支持普通上传/覆盖
  - 仅支持下载补传
  - 当前未实现
- `秒传 JSON 直导` 继续只对 Guangya 显示，并明确写清“当前仅 Guangya 支持”

### 10. 配置结构与持久化

- 扩展 `config.py` 的分组结构，至少拆开保存：
  - 外部本地 OpenList 配置
  - 外部远程 OpenList 配置
  - 托管二进制配置
  - 托管初始化管理员配置
  - 控制台管理员配置
  - 当前任务配置
  - UI 选择与目录浏览状态
- 旧配置继续兼容读取
- 重新进入页面时，目录选择和 UI 状态要按配置恢复，不再要求用户手动退一级再刷新

### 11. 文档统一

- 更新 `README.md`
- 统一对外叙事为：
  - CloudPan Bridge 是“统一源端 -> 多目标端”的桥接控制台
  - OpenList 负责源端接入与挂载
  - Guangya 是当前秒传能力最完整目标端
  - 托管模式优先是“本机二进制托管”
- 新增一节明确说明：
  - 外部模式与托管模式的区别
  - 当前托管为什么需要本机二进制
  - 若本机没有 OpenList 二进制时，会先提示是否拉取
  - 如果不拉取，只能改用远程/外部模式
  - Docker 托管当前仅预留，不是本轮正式能力
  - 若本机没有 Docker，如何安装、启动并完成最小验证

## 接口与行为变化

- 保留已有主要接口路径，优先兼容现有前端逻辑
- 新增或重构的接口语义包括：
  - 托管运行时预检
  - 托管运行时状态详情
  - 托管二进制缺失时的拉取建议/确认所需状态
  - 模式独立配置读写
  - 目标端能力预检
- 本轮不引入“自动 Docker 托管启动 OpenList 容器”的真实后端逻辑，只保留文案边界和状态占位，避免误导用户

## 测试计划

- 配置与页面
  - 未登录控制台时，主操作区不可见或不可交互
  - 登录后显示统一主流程
  - 外部本地 / 外部远程 / 托管二进制三种模式切换后，各自配置独立保存并恢复
- 托管 OpenList
  - 本机无二进制时：
    - 页面明确提示是否拉取
    - 日志记录原因
    - 不再静默失败
  - 用户拒绝拉取时：
    - 托管模式不启动
    - 页面提示改用远程/外部模式
  - 本机有二进制时：
    - 能显示最终二进制路径、端口、访问地址
    - 能成功探活
- Docker 环境预检
  - 本机无 Docker 时：
    - 页面明确提示需要先安装 Docker
    - README 提供安装与验证步骤
    - 不误判为 Docker 可用
  - 本机有 Docker 时：
    - 能正确识别 Docker 可用状态
    - 能进入后续 Docker 托管预留链路的状态检查
- 外部 OpenList
  - 本机已有实例可登录并写回 token
  - 远程实例可连接、列目录、读挂载列表
- 托管接受/拒绝决策矩阵
  - 本机二进制缺失，用户拒绝拉取：
    - 托管模式终止
    - 明确提示改用外部/远程模式
  - 本机二进制缺失，用户接受拉取：
    - 完成下载
    - 再次预检
    - 成功后允许启动托管
  - Docker 不可用或用户拒绝进入 Docker 相关链路：
    - 明确提示安装或切换模式
    - 不进入虚假的“已托管”状态
  - Docker 可用且用户接受进入 Docker 相关链路：
    - 至少完成可用性识别与后续状态入口验证
    - 为后续真正 `managed_docker` 落地预留验证基线
- 源端流程
  - 已挂载源可选目录并写入 `source_path`
  - 未挂载源可在同一主流程里完成接入并回选
- 目标端流程
  - 目标端切换时字段区与说明区同步切换
  - Guangya 显示秒传/直导入口
  - 非 Guangya 不显示误导性的秒传入口
- 文档与 CI
  - README 与页面行为一致
  - GitHub Actions 不再在 `push` 时自动构建/推送 Docker 镜像

## 假设与默认值

- 本轮默认优先支持 Windows 桌面本机使用场景
- 本轮托管优先实现 `managed_binary`
- `managed_docker_placeholder` 只做结构预留和文案边界，不承诺本轮真正落地
- 若本机没有 OpenList 二进制，默认行为不是直接报错结束，而是优先提示“是否拉取”；若用户不拉取，再引导改用远程/外部模式
- 默认同步语义继续保持“新增 + 覆盖”，删除仍是独立开关，不作为默认行为
